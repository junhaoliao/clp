fix(helm): Replace results-cache-indices-creator Job with native sidecar for replica set re-initialization (fixes #2257).

# Description

When the results-cache MongoDB's PersistentVolumeClaim (PVC) is deleted and recreated (e.g., after an EKS node replacement with an availability zone mismatch), MongoDB starts with a fresh data directory but retains `replSetName: "rs0"` in its config. This puts MongoDB in an RSGhost state — it expects to be part of a replica set but has no initialized configuration. All clients then fail with `ServerSelectionTimeoutError: No servers match selector 'Primary()'`.

The root cause is that the `results-cache-indices-creator` Kubernetes Job, which handles replica set initialization and index creation, only runs once on initial deployment. After a PVC recreation, the completed Job doesn't re-run, leaving MongoDB stuck in RSGhost state.

This change replaces the one-time Job with a **native sidecar** (K8s 1.28+ init container with `restartPolicy: Always`) in the results-cache StatefulSet:

- Adds a `results-cache-indices-creator` sidecar init container using the CLP package image, which runs `clp_py_utils.initialize-results-cache` (the same Python script used by Docker Compose deployments) on every pod start. If the script fails (e.g., MongoDB isn't ready yet), the sidecar restarts and retries. After the script succeeds, the sidecar sleeps indefinitely.
- Modifies the readiness probe to check for PRIMARY availability (not just ping), ensuring dependent services don't connect until the replica set is fully functional.
- Keeps the liveness probe as a simple ping check to avoid container restart loops during replica set transitions.
- Gates the `results-cache-indices-creator` Job to only run when results_cache is **not** bundled (i.e., externally-managed MongoDB). In this case, the Job is the only mechanism to initialize the replica set and create indexes.
- Adds a `clp.waitForResultsCache` helper template that conditionally renders the correct `waitFor` init container: waits for the `results-cache` service when bundled, or the `results-cache-indices-creator` Job when not bundled.
- Updates 5 dependent deployments (webui, reducer, garbage-collector, api-server, mcp-server) to use the new `clp.waitForResultsCache` helper.

Docker Compose deployments are unaffected — the `results-cache-indices-creator` service already re-runs on every `docker compose up`, correctly handling re-initialization.

# Impact Assessment

- **Helm/K8s deployments with bundled results_cache**: The `results-cache-indices-creator` Job is replaced by a native sidecar in the MongoDB StatefulSet pod. Requires Kubernetes 1.28+ for native sidecar support.
- **Helm/K8s deployments with external MongoDB**: The `results-cache-indices-creator` Job still runs once to initialize the replica set and create indexes (no change from current behavior).
- **Docker Compose deployments**: No changes. The existing `results-cache-indices-creator` service continues to work as before.
- **Readiness probe**: The MongoDB readiness probe now requires a PRIMARY to be available, which may cause slightly longer startup times for first-time deployments (typically 5-10 seconds for primary election). This is the correct behavior — dependent services should not connect until the replica set is functional.
