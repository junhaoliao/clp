‚óŹ Grafana's approach:

  Grafana has two query patterns:
  1. Standard query() ‚ÄĒ The datasource plugin blocks until results are ready,
  then returns DataFrame[]. For async backends (like Loki, InfluxDB), the plugin
   internally polls/waits and returns complete results. The panel shows a
  spinner until the full response arrives.
  2. Streaming/Live data (LiveDataFrame) ‚ÄĒ Introduced for real-time use cases.
  The datasource returns a "streaming" frame with a channel key. Grafana
  subscribes to that channel (via WebSocket) and incrementally appends new rows
  to the frame. The panel re-renders as new data arrives. This is how Loki
  live-tail and Prometheus real-time dashboards work.

  The key insight: Grafana's LiveDataFrame is still a DataFrame ‚ÄĒ it just has a
  channel field that tells the frontend "keep listening for more rows." The
  schema (field names, types) is fixed from the first frame; only values[]
  arrays grow.

  Metabase's approach:

  Metabase uses a simpler job-based pattern:
  1. POST creates a query job, returns { id }
  2. Client polls GET /api/query/:id until status = completed
  3. Full results returned as a single response when done
  4. Dashboard cards show loading spinners independently; a worker pool limits
  concurrent queries (5 for HTTP/1.1)
  5. No SSE ‚ÄĒ no progressive results. The tradeoff is simplicity over latency.

  For CLP:

  The CLP query engine is inherently async (job queue ‚Üí Celery workers ‚Üí MongoDB
   results). The existing webui already uses Socket.IO to push incremental
  results from MongoDB change streams. Two viable approaches:

  Option A: SSE + Job Queue (recommended)
  - POST /api/datasource/clp/query submits a search job via QueryJobDbManager,
  returns { jobId }
  - Open SSE stream at /api/datasource/clp/query/stream?jobId=xxx
  - Server reads MongoDB results incrementally (via cursor or change stream),
  converts batches to DataFrame partial events
  - SSE events: { type: "schema", fields: [...] } then { type: "partial", data: 
  DataFrame } then { type: "complete", data: DataFrame }
  - Client accumulates partial DataFrames into a single frame (same pattern as
  MySQL SSE but with async polling for job completion)

  This works because the DataFrame schema is deterministic for CLP queries ‚ÄĒ KQL
   search results have fixed columns (timestamp, message, file, log_event_id,
  etc.). The values[] arrays just grow as results stream in.

  Option B: Polling (Metabase-style)
  - Submit job, return job ID
  - Client polls GET /api/datasource/clp/query/:jobId every 2s
  - Returns full DataFrame when job completes, partial progress metadata
  otherwise
  - Simpler but higher latency ‚ÄĒ no progressive rendering

  Option A is strictly better for CLP because: (1) search queries can return
  millions of results ‚ÄĒ streaming is essential, (2) the existing Socket.IO
  infrastructure already does incremental push, and (3) the MySQL SSE pattern is
   already working in the dashboard.

‚óŹ Now the client-side change ‚ÄĒ executePanelQuery needs to use SSE for CLP
  queries to get progressive rendering. Let me check how the MySQL streaming is
  consumed on the client side, if at all.

  The client currently calls /api/datasource/${dsType}/query (non-streaming) for
   all datasource types. For CLP, we need it to use
  /api/datasource/clp/query/stream with SSE consumption, because the job-queue
  pattern can take seconds and partial DataFrames give progressive rendering.

  However, I need to think about this carefully. The current usePanelQueries
  uses @tanstack/react-query which expects a Promise<DataQueryResponse>. SSE
  consumption is a fundamentally different pattern ‚ÄĒ it's an event stream, not a
   single promise resolution.

  Two options:
  1. Make executePanelQuery return a Promise that resolves once the SSE stream
  completes (accumulate partials internally)
  2. Change the hook to support streaming

  Option 1 is much simpler ‚ÄĒ the Promise resolves with the final merged
  DataFrame once streaming is complete, but partial DataFrames are used to
  update the UI progressively via a callback or state update. Actually, looking
  more carefully at usePanelQueries, the queryFn returns
  Promise<DataQueryResponse> ‚ÄĒ the entire response must resolve before the panel
   renders.

  For progressive rendering, we'd need to update the data state incrementally
  during streaming. This is a more significant change. Let me go with a
  pragmatic approach: use SSE on the client but accumulate into a single
  response. Progressive rendering can be added later.
